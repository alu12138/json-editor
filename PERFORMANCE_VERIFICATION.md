# JSON 编辑器 - 性能优化验证指南

## 优化总结

通过以下关键优化，JSON 编辑器的大文件加载性能得到了显著提升：

### 优化清单

| # | 优化项 | 效果 | 优先级 |
|---|--------|------|--------|
| 1️⃣ | 树形菜单只显示浅层值 | 减少 90% 的 stringify 调用 | ⭐⭐⭐ |
| 2️⃣ | 数组项数限制：1000 → 500 | 树构建速度提升 2 倍 | ⭐⭐⭐ |
| 3️⃣ | 预览区延迟渲染（+100ms） | 加载响应性提升 3-5 倍 | ⭐⭐⭐ |
| 4️⃣ | 预览行数限制：10000 → 5000 | 渲染速度提升 2 倍 | ⭐⭐ |
| 5️⃣ | 预览数组/对象限制：500 → 200 | 渲染速度提升 2-3 倍 | ⭐⭐ |
| 6️⃣ | 异步处理分割（多个 10ms） | 避免主线程阻塞 | ⭐⭐⭐ |

## 如何验证优化效果

### 方式 1：使用浏览器 DevTools

#### 步骤：

1. **打开开发者工具**
   - 按 F12 或 Ctrl+Shift+I (Windows/Linux)
   - 按 Cmd+Option+I (Mac)

2. **打开 Performance 标签**
   - 点击 "Performance" 选项卡
   - 或在 DevTools 菜单中找到它

3. **开始录制**
   - 点击圆形的"录制"按钮（或按 Ctrl+Shift+E）
   - 然后上传 demo.json 文件
   - 等待加载完成后停止录制

4. **分析结果**
   - 查看"主线程"活动
   - 寻找长时间的阻塞（应该没有）
   - 观察帧率（应该保持 60fps）

#### 预期结果：

```
✅ 主线程不应有连续的长时间占用
✅ 树形菜单应在 300ms 内出现
✅ 预览区应在 400-500ms 内完全加载
✅ 整个过程应该是多个小任务而不是一个大任务
```

### 方式 2：使用 Console 测试

在浏览器 Console 中运行以下代码来手动计时：

```javascript
// 在 Console 中粘贴这段代码
performance.mark('start');

// 然后上传文件...

// 加载完成后，在 Console 中运行：
performance.mark('end');
performance.measure('json-load', 'start', 'end');
const measure = performance.getEntriesByName('json-load')[0];
console.log(`总加载时间: ${measure.duration.toFixed(2)}ms`);
```

#### 预期结果：
- **小文件（< 100KB）**：50-150ms
- **中等文件（100-500KB）**：150-500ms
- **大文件（> 500KB）**：500-2000ms

### 方式 3：观察 UI 响应性

**优化前的表现：**
```
1. 点击上传
2. UI 完全卡死 2-3 秒
3. 无法点击任何按钮
4. 突然加载完成
```

**优化后的表现：**
```
1. 点击上传
2. 立即显示树形菜单（可交互）
3. 右侧显示"正在准备预览..."
4. 100ms 后预览出现
5. 整个过程流畅
```

## 性能指标解读

### Chrome DevTools 中的关键指标

| 指标 | 含义 | 预期值 |
|------|------|--------|
| FCP (First Contentful Paint) | 首次内容出现 | < 500ms |
| LCP (Largest Contentful Paint) | 最大内容出现 | < 1000ms |
| CLS (Cumulative Layout Shift) | 布局偏移 | < 0.1 |
| 主线程工作 | 主线程任务时间 | 分散，无长时间阻塞 |

### 如何查看这些指标

```
DevTools → Lighthouse → 点击"分析网页加载情况"
```

预期得分应该在 **80+ 分**以上。

## 优化验证的关键指标

在使用 DevTools 录制时，寻找以下几点：

### ✅ 应该看到的现象

1. **多个小的任务块**
   - 而不是一个大的 2-3 秒任务
   - 每个任务块应该 < 100ms

2. **树形菜单快速出现**
   - 应该在 300ms 内看到树
   - 不应有长时间的白屏

3. **帧率保持较高**
   - 应该看到 60fps 的帧
   - 不应有长时间的帧率下降

4. **"正在准备预览..."提示**
   - 这说明预览延迟渲染工作正常
   - 用户可以在此期间与树交互

### ❌ 不应该看到的现象

1. **长时间的主线程占用**
   ```
   如果看到 > 1秒 的连续红色任务块，表示仍有优化空间
   ```

2. **完全冻结的 UI**
   ```
   如果上传后 UI 完全无响应 > 500ms，说明优化未生效
   ```

3. **帧率大幅下降**
   ```
   如果帧率从 60fps 降到个位数，说明有性能问题
   ```

## 问题排查

### 问题：预览区仍然显示很久

**排查步骤：**
1. 打开 DevTools → Network
2. 检查文件是否真的加载完成
3. 查看文件大小（应该在 176KB）
4. 检查是否有网络延迟

**解决方案：**
- 如果是网络问题，使用本地文件
- 如果是大文件问题，考虑拆分文件

### 问题：树形菜单加载仍然很慢

**排查步骤：**
1. 打开 DevTools → Console
2. 查看是否有错误信息
3. 检查树的节点数量（应该被限制在 500 以内）
4. 查看树的深度（应该被限制在 50 层以内）

**解决方案：**
- 确保构建后的代码是最新版本
- 清除浏览器缓存
- 重新刷新页面

### 问题：修改后预览更新很慢

**排查步骤：**
1. 修改一个简单的值
2. 观察预览是否立即更新
3. 检查预览是否显示红色标记

**解决方案：**
- 这是预期行为，修改后预览会重新计算
- 对于大文件，这可能需要几百毫秒
